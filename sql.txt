INFORMATION_SCHEMA.TABLES
INFORMATION_SCHEMA.TABLES is a system view provided by most relational databases 
Purpose: Give you a consistent way to query database metadata (tables, columns, constraints, etc.) without relying on vendor-specific system tables.

INFORMATION_SCHEMA.TABLES contain :

TABLE_CATALOG → Database name
TABLE_SCHEMA → Schema name (e.g., dbo, prc)
TABLE_NAME → Table name
TABLE_TYPE → 'BASE TABLE' (real table) or 'VIEW'

lock in postgress
Are you going to UPDATE or DELETE the row?
│
├── YES
│    ├── Will you change the primary/unique key or DELETE?
│    │    └── YES → FOR UPDATE 
│    │
│    └── Only changing non-key columns (balance, status, name...)?
│         └── FOR NO KEY UPDATE  ✅ preferred ,it allows fk check lock as for key share 
│              
└── NO (just reading)
     ├── Need to ensure row won't be modified while you read?
     │    └── FOR SHARE
     │
     └── Just need the row's key/identity to stay (FK check)?
          └── FOR KEY SHARE  (PostgreSQL does this automatically)
              this lock insures that fk in current transaction exist as pk of other table , it is not delelted, locks this pk

for no key update : this lock is very optimal coz it locks the row , whoes values are getting updated nd deleted (it insures only non key values are getting modified or deleted)
it allows FOR KEY SHARE lock while current row is in lock of FOR NO KEY UPDATE lock , ususally FOR KEY SHARE is used by postgress to insure that keys which are used in that transaction are exists nd if exists it locks that keys so that it wont be updated or deleted

for update : bt here there is strict lock on row , FOR KEY SHARE is also not allowed here , this used when we have to change fk nd pk also 

shared lock means it aquires lock and other can also aquired shared lock , means other also can read bt no one can modify
* note : in mssql any select statment held shared lock automatically, means while selecting other can read bt no one can update 
        thats why we use with(nolock) in select which allows to update , and select can dirty read the data
NOWAIT: don’t wait—error immediately if the row is locked
eg.
SELECT *
FROM jobs
WHERE id = 7
FOR UPDATE NOWAIT;

Plain SELECT always runs instantly regardless of any locks. It just reads the last committed snapshot. This is the biggest practical benefit of MVCC — readers never block writers, writers never block readers.

Tip	                   Problem it solves
Consistent lock order	 Two transactions waiting on each other in a cycle
Short transactions	   Locks held too long, queuing up other transactions
Selective WHERE (PK)	 Accidentally locking rows you don't need
